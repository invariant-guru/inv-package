# Skill: Create Invariant Package

Scaffold a new Invariant package вЂ” directory structure, item files, and a single `invariant.json` manifest вЂ” then let the CLI populate and validate it.

## Usage

Invoke when a user wants to author a new publishable Invariant package. Gather metadata, create the item files on disk, then drive the `invariant package` CLI to generate the manifest.

## Key facts

- One manifest file: **`invariant.json`**. (The older `invariant-package.json` no longer exists.)
- The same `invariant.json` schema powers three modes, auto-detected by content:
  - **Project** вЂ” consumes packages (`targets` / `packages` / `active`).
  - **Package** вЂ” publishable (`items` block present).
  - **Hybrid** вЂ” both.
- The `items` block is **auto-generated by scanning the package directory** вЂ” you never hand-write it. Create the files first; the CLI fills `items` in and extracts each item's description from the first non-empty line of its primary file.

## The manifest (package mode)

```json
{
  "name": "clean-arch",
  "version": "1.2.0",
  "description": "Clean architecture rules and agents.",
  "author": "invariant-labs",
  "license": "MIT",
  "repository": "https://github.com/invariant-labs/clean-arch",
  "items": {
    "agents": [{ "name": "planner", "description": "Plans tasks by layer." }],
    "rules":  [{ "name": "layer-deps", "description": "Enforces dependency direction." }]
  }
}
```

| Field | Required | Notes |
|---|---|---|
| `name` | yes | lowercase, hyphens allowed; the `invariant install <name>` id |
| `version` | yes | semver `major.minor.patch` |
| `description` | recommended | shown in registry / search |
| `author` | no | name or handle |
| `license` | no | e.g. `MIT`, `Apache-2.0` |
| `repository` | no | source repo URL |
| `items` | auto | `type в†’ [{name, description}]`. Presence marks the file a package. **Generated by `invariant package create` / `refresh` вЂ” do not hand-maintain.** |

## Item types (exactly six)

`agents`, `skills`, `commands`, `rules`, `contexts`, `instructions`.

- **Skills** вЂ” always a folder `skills/<name>/` with a required `SKILL.md`; extra `.md` files and sub-folders are bundled. Item name = folder name. A skill is never a single file.
- **All other types** вЂ” either a single file `<type>/<name>.md` (item name = filename without `.md`) or a folder `<type>/<name>/*.md` (item name = folder name; all `.md` bundled). Mix freely within a type.
- Only include directories for the types your package provides.

## Procedure

### Step 1 вЂ” Gather metadata

Ask for: package **name** (lowercase, hyphens), **description**, **author**, **license** (default `MIT`), **repository** (optional), initial **version** (default `1.0.0`).

### Step 2 вЂ” Choose items

Ask which of the six types are needed; for each item collect a **name** and a one-line **description**. The description should become the first heading/line of the item file вЂ” the CLI reads it back into the manifest.

### Step 3 вЂ” Scaffold the files on disk

Create the package directory and, for each item:
- **Skill** в†’ `skills/<name>/SKILL.md` (+ any support files).
- **Other** в†’ `<type>/<name>.md` (single file) or `<type>/<name>/*.md` (folder).

Author each file's content (see guidelines below). **Do not hand-write the `items` block** вЂ” the CLI generates it next.

### Step 4 вЂ” Generate the manifest

From the package root:

```bash
invariant package create \
  --name <name> --version <version> \
  --description "<desc>" --author "<author>" \
  --license MIT --repository "<url>"
```

This scans the folder, writes/updates `invariant.json`, and fills the `items` block (it runs a refresh internally). If `invariant.json` already exists it is updated in place and switched to Package (or Hybrid) mode.

After adding/removing item files later, re-sync the manifest:

```bash
invariant package refresh
```

### Step 5 вЂ” Validate

```bash
invariant package validate
```

Reports `missing_on_disk` (declared but file absent) and `undeclared_on_disk` (file present but not in manifest). Resolve drift вЂ” usually by running `invariant package refresh`.

### Step 6 вЂ” Publish

```bash
invariant package publish
```

Validates, bundles (gzip tar of `invariant.json` + item files), and uploads to the registry. Requires authentication: set `INVARIANT_TOKEN`, or run `invariant login`.

## Content guidelines per type

- **Agents** вЂ” `## Role`, `## Approach`, `## Workflow`.
- **Skills** вЂ” `SKILL.md` with `## Usage` (when to invoke) + `## Procedure` (ordered `### Step N`).
- **Commands** вЂ” `## Trigger`, `## Parameters`, `## Behavior`.
- **Rules** вЂ” constraints as must / must-not bullet points.
- **Contexts** вЂ” background knowledge + reference material.
- **Instructions** вЂ” embedded into AI context every interaction:
  1. Open with a **"This Package Contains"** table (item, type, when to use) + a partial-install note.
  2. Self-contained but concise.
  3. Reference any related skill explicitly.
  4. Example-driven (short fenced blocks).
  5. Action-first.
  6. End with a `- [ ]` checklist.

Instruction template:

````markdown
# <Topic>

<What this covers.> For interactive guidance, invoke the **`<skill>`** skill.

## This Package Contains

| Item | Type | When to use |
|------|------|-------------|
| `<instruction>` | Instruction | <short description> |
| `<skill>` | Skill | <short description> |

> **Partial installs:** Some items above may not be present in your
> `.claude` folder. Invariant lets users install only a subset of a
> package. Use what's available.

## <Section>

<Concise explanation + a short fenced example.>

## Checklist

- [ ] Validation point 1.
- [ ] Validation point 2.
````

## Checklist

- [ ] Manifest is `invariant.json` (never `invariant-package.json`).
- [ ] `name` lowercase + hyphens; `version` is semver.
- [ ] `items` block was generated by the CLI (`package create` / `refresh`), not hand-written.
- [ ] Every skill is a folder containing `SKILL.md`.
- [ ] Other items are single `.md` files or folders of `.md` files.
- [ ] `invariant package validate` reports no drift.
- [ ] Authenticated (`INVARIANT_TOKEN` / `invariant login`) before `invariant package publish`.
